AIT10: When My Website Stopped Updating —
A Vercel Deployment Freeze AIT10：当我的网站停止更新时——
一次Vercel部署卡死事件

📅 July 20262026年7月 ⏱️ 6 min read6分钟阅读
✅ Link copied!链接已复制！

I kept asking Claude to add things to my Table 30 options page. Claude kept saying "done, pushed, merged." But the live site just... didn't change. Here's what actually happened, and the exact steps I'm keeping for next time. 我一直让Claude在我的Table 30期权页面上添加内容。Claude一直说「完成了，已推送，已合并」。但实际网站却……毫无变化。这是真正发生的事情，以及我为下次保留的具体步骤。

What Happened发生了什么

Claude had made a whole string of changes to my Options Breadboard page — a new Q&A, a sorted trade log, a brand-new reflection article. Every single time, it told me the change was committed to GitHub and merged into the live branch. But when I opened the actual link on my phone, I kept seeing the old version — or a flat 404 error for a page that should have existed.Claude对我的期权面包板页面做了一连串更改——新的问答、排序后的交易记录、一篇全新的反思文章。每一次它都告诉我更改已提交到GitHub并合并到了正式分支。但当我在手机上打开实际链接时，看到的却一直是旧版本——或者是本该存在的页面出现404错误。

I refreshed. I tried Private Browsing. I cleared cache. Nothing changed. Something was stuck between "Claude says it's done" and "my phone can actually see it."我刷新过，试过隐私浏览模式，也清过缓存。什么都没变。在「Claude说完成了」和「我的手机真的能看到」之间，有什么东西卡住了。

What It Wasn't这不是什么原因

Before assuming the worst, we ruled things out one by one:在假设最坏情况之前，我们逐一排除了各种可能：

Things we checked and confirmed were fine我们检查并确认没问题的事项
  • GitHub had every single change — confirmed directly, every time, by fetching the file straight from GitHub's own servers.GitHub上确实有每一处更改——每次都直接从GitHub自己的服务器上核实过。
  • The website's own code was fine — no broken config file, no setting telling Vercel to skip builds.网站自身的代码没有问题——没有损坏的配置文件，也没有设置让Vercel跳过构建。
  • The connection between GitHub and Vercel (the "app" that watches for new changes) was installed correctly, with no warning or permission issue.GitHub与Vercel之间的连接（负责监视新更改的「应用程序」）安装正确，没有任何警告或权限问题。
  • When a build DID run, it finished successfully in about 25 seconds every time — so it wasn't a broken build either.当构建确实运行时，每次都能在约25秒内成功完成——所以也不是构建本身出了问题。

What It Actually Was真正的原因

Here's the part that's genuinely useful to understand: publishing a change to a website like mine has two separate steps, and they can fail independently.这部分才是真正值得理解的：像我这样的网站要发布一次更改，其实有两个独立的步骤，而它们可能各自出问题。

Step A — Save the change (GitHub)步骤A——保存更改（GitHub）

This is Claude writing the file and saving it. This part never failed — every change was safely saved, the whole time.这是Claude编写文件并保存的过程。这部分从未失败——所有更改始终都被安全保存着。

Step B — Publish the change (Vercel)步骤B——发布更改（Vercel）

A separate company (Vercel) hosts the actual live website. It's supposed to notice every time GitHub is updated and automatically rebuild the site. For about an hour, it stopped noticing — most updates went in one ear and out the other, with no error message anywhere. Then, just as suddenly, it caught up and started working again on its own.另一家公司（Vercel）负责托管实际的正式网站。它本该每次GitHub更新时都察觉到，并自动重新构建网站。有大约一个小时的时间，它「没有察觉」——大多数更新都石沉大海，没有任何错误提示。然后，突然间，它又自己恢复正常了。

The code was never broken. The message just got lost between two computers talking to each other — and I had no way to see that from my side until we went looking for it.代码从来没有出过问题。只是两台电脑之间的消息在传递途中丢失了——而在我们去查找之前，我从自己这边根本无法察觉这一点。

My Checklist For Next Time下次遇到时的检查清单

Don't panic — wait 30-60 minutes first别慌——先等30-60分钟

It resolved itself last time without anyone fixing anything. Most of the time, patience alone is the fix.上次它是自己恢复的，没有任何人做任何修复。大多数情况下，耐心等待本身就是解决办法。

Tell Claude right away — it can check GitHub instantly立即告诉Claude——它能立刻查GitHub

Claude can confirm in seconds whether my content actually saved correctly, before we ever assume it's a bigger problem.Claude能在几秒内确认我的内容是否确实正确保存了，无需一开始就假设是更大的问题。

Still stuck after an hour? Use the computer, not the phone一小时后仍卡住？用电脑，别用手机

The Vercel dashboard is much easier to see clearly on a real screen. I use TeamViewer to reach my computer remotely when I'm travelling without my laptop.Vercel控制台在真正的大屏幕上看起来清楚得多。出门在外没带笔记本电脑时，我会用TeamViewer远程连接我的电脑。

Look at the "Deployments" list for the project查看项目的「Deployments」（部署）列表

If the newest entry is old, and nothing newer shows up at all — not even as an error — that's the same signature as last time. It's their side, not mine.如果最新的一条记录是旧的，而且完全没有更新的记录出现——甚至连错误提示都没有——那就和上次的情况一模一样。是他们那边的问题，不是我这边的。

If it's been over an hour, email support — in the same thread as last time如果超过一小时，发邮件给客服——回复上次同一封邮件

I already have a draft explaining exactly what to look for. Reply to that thread instead of starting fresh, so they have the history.我已经有一份草稿，准确说明了该检查什么。回复那封邮件的原有对话，而不是重新写一封，这样对方就能看到历史记录。

⚠️

One important limit: Claude cannot see the live Vercel dashboard or fix a stuck deployment by itself — it can only confirm GitHub is correct, and guide me through the Vercel screens. The actual click-through has to be me.一个重要的限制：Claude无法自己查看Vercel的实时控制台，也无法自行修复卡住的部署——它只能确认GitHub那边没问题，并指导我操作Vercel的界面。真正点击操作的必须是我自己。

✅

The reassuring part: nothing was ever lost. Every single change was safely saved on GitHub the whole time — this was purely a "publishing" delay, not a "losing my work" problem.让人安心的一点：没有任何东西丢失。每一处更改在GitHub上始终被安全保存着——这纯粹是「发布」延迟的问题，不是「丢失我的工作成果」的问题。
