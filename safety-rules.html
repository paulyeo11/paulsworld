Safety Rules & Precautions

Important rules to protect your website, GitHub files, and images.
Read this before making ANY changes to your website.

Why This Page Exists

Files were accidentally deleted from GitHub twice — including entire image folders. Restoring from backup was difficult and all image folders were lost. This page lists every precaution to prevent this from happening again.

🚫

NEVER Delete Files on GitHub

The #1 most dangerous action — do NOT do this

  • ❌NEVER delete any file or folder directly on GitHub.com using the web interface — not even "old" or "unused" files.
  • ❌NEVER delete an image folder (e.g. img_macritchie/, img_swiss/). Deleting the folder removes ALL photos inside permanently.
  • ❌NEVER use "Delete this file" from the GitHub pencil/edit menu unless you are 100% certain and have a local backup.
  • ✅If you want to remove a page, Archive it instead — rename it by adding "_old" at the end (e.g. s3_old.html) so it still exists but is no longer linked.
  • ✅Before deleting anything, always ask yourself: "Do I have a complete local copy of this?" If not — DO NOT delete.
🖼️

Always Back Up Image Folders

Images are the hardest to recover — protect them first

  • 📁Every article has its own image folder (e.g. img_macritchie/, img_swiss/, img_portugal/). Keep a copy of ALL these folders on your local computer AND external hard drive.
  • 📸After uploading new photos to GitHub, immediately copy them to your local backup folder. Don't wait.
  • 🗂️Keep a master list of all image folder names so you know what exists. Current folders: img_macritchie, img_swiss, img_portugal — update this list whenever you add new ones.
  • ☁️Consider also backing up image folders to Google Drive or Dropbox as a second cloud backup (separate from GitHub).
  • ⚠️Vercel restore (backup) does NOT restore image folders — only HTML files. You must restore images manually. This is what caused the problem before.
🔍

Before Making Any Website Change

Always follow this checklist before editing

  • 1️⃣Take a screenshot of your GitHub repository file list so you know what files exist before you start.
  • 2️⃣If editing an HTML file, copy the current version to your computer first (right-click → Save As) before making changes.
  • 3️⃣When AI pushes a file to GitHub for you, confirm the push was successful before closing the chat. Always check the commit SHA that was returned.
  • 4️⃣After any AI push, visit the live page on paulsworld.vercel.app to confirm it displays correctly.
  • 5️⃣Never do two big changes at the same time. One change at a time, confirm it works, then do the next one.
💾

Your Backup Strategy (3 Copies Rule)

Always have 3 copies: GitHub + Local + Cloud

  • 🔵Copy 1 — GitHub (Live): Your main website files at github.com/paulyeo11/Dynamic-Index. This is your "production" copy.
  • 🟢Copy 2 — Local Computer: Keep a folder on your desktop called paulsworld_backup with ALL HTML files AND all image folders. Update it weekly.
  • 🟡Copy 3 — Google Drive / USB: Upload your entire paulsworld_backup folder to Google Drive once a month. Also keep a copy on a USB stick.
  • 📅Set a monthly reminder (every 1st of the month) to do a full backup — download all files from GitHub and save to local + Google Drive.
  • 🆕Every time a NEW article or page is created, immediately add its images to your local backup. Don't wait for the monthly backup.
🤖

Safe Rules When Using AI (Claude)

How to use AI safely to manage your website

  • ✅When AI pushes a file, always ask AI to confirm the commit SHA (the long code like abc123def). This proves the file was saved correctly.
  • ✅Tell AI to NEVER delete files — only create new or update existing ones. Make this a standing rule in every session.
  • ✅After AI updates index.html, always check the live website immediately to confirm all sections still appear correctly.
  • ⚠️If AI shows you an error or unexpected result, STOP and ask AI to explain before continuing. Never keep going blindly when something looks wrong.
  • ⚠️Keep your AI conversations focused. One task per session if the task involves changing multiple files — rushing leads to mistakes.
  • 🔒Your GitHub token (ghp_...) is private and powerful — it can modify ALL your files. Never share it publicly or paste it in chat messages to other people.
📂

File Naming & Organisation Rules

Keep everything neat and easy to find

  • 📝All HTML files follow a prefix system: s1–s7 (Stories), ai1–ai4 (AI Journey), ac1–ac2 (Activities), book1–book2 (Books), h1–h10 (Health), travel1 (Travel).
  • 🖼️Every article's image folder must be named img_[articlename] (e.g. img_macritchie, img_swiss). Never put images loose in the root folder.
  • 🚫Never use spaces in file or folder names. Use underscores only (e.g. img_new_zealand not img new zealand).
  • 🔤Use lowercase letters only for all file and folder names. GitHub is case-sensitive — Img_Swiss and img_swiss are different folders.
  • 📋Keep a simple text file called my-files-list.txt on your desktop listing every file and folder that exists on GitHub. Update it whenever something is added.

🆘 If Something Goes Wrong — Recovery Steps

1
STOP — don't panic, don't make more changes Stop all editing immediately. Making more changes when something is broken makes recovery harder.
2
Check GitHub commit history Go to github.com/paulyeo11/Dynamic-Index → click "Commits" → find the last good version → click the file → click "Raw" to copy the old content.
3
Restore HTML files first HTML files can be restored from GitHub history or your local backup. Tell AI: "Please restore this file from my backup" and paste the old content.
4
Restore image folders from LOCAL backup ⚠️ Vercel/GitHub history does NOT include image folders. You MUST restore images from your local computer or Google Drive backup. Re-upload each image folder to GitHub manually.
5
Test the live website thoroughly Visit every page on paulsworld.vercel.app and check all images load correctly. Check the index page links all work.
