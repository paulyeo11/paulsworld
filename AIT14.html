AIT14: The Frozen Deploy —
Finding the One Button That Fixed It AIT14：卡住的部署——
找到那个一键修好的按钮

📅 24 July 20262026年7月24日 ⏱️ 5 min read5分钟阅读
✅ Link copied!链接已复制！

I asked for a new article about coin tosses and trading. Claude wrote it, pushed it, and told me to check the link. I got a 404. Twenty minutes and a dozen screenshots later, we found the exact button that fixed it — and this time it took minutes, not the two-hour migration from last time. 我请Claude写一篇关于抛硬币与交易的新文章。Claude写好、推送了，让我去看链接。结果我看到的是404错误。二十分钟、十几张截图之后，我们找到了那个一按就能解决问题的确切按钮——而这一次只花了几分钟，不像上次那样折腾了两个小时才搬家。

A New Article, an Old Ghost一篇新文章，一个老幽灵

This one started simply: I asked for an article on the coin-toss-and-trading idea I'd been thinking about — small samples lie, large samples tell the truth, and the real edge comes from keeping wins bigger than losses. Claude built it as T49, with a live coin-flip simulator right on the page, added it to my Investment Tools hub, committed it, and pushed it straight to the live site like normal.这次事情很简单：我请Claude写一篇我一直在想的「抛硬币与交易」的文章——小样本会骗人，大样本才说真话，真正的优势来自让盈利始终大于亏损。Claude把它做成了T49，页面上还带了一个实时抛硬币模拟器，加进了我的投资工具中心，提交、然后像平时一样直接推送到了正式网站。

Then I tapped the link. 404 — page not found.然后我点了链接。404——页面找不到。

Same ghost as AIT10 and AIT12. Vercel's production deployment had quietly stopped picking up new pushes to my site — again.和AIT10、AIT12里的幽灵一模一样。我网站的Vercel正式部署又一次悄悄不再接收新的推送了。

Ruling Out the Easy Answers First先排除简单的可能

Before assuming the worst, Claude checked the boring stuff first: was the file actually saved? Was it actually pushed to the right branch? A quick check confirmed the new page was sitting safely on GitHub's main branch, exactly where it needed to be. Nothing was lost — the problem wasn't the code, it was getting Vercel to notice it.在往最坏的方向想之前，Claude先排查了些基本问题：文件真的存好了吗？真的推送到正确的分支了吗？一查，确认新页面确实安全地躺在GitHub的main分支上，位置完全正确。什么都没丢——问题不在代码本身，而在于怎么让Vercel注意到它。

Screenshots, Again再一次，靠截图

Claude still can't see my Vercel dashboard — that hasn't changed since last time. So we went back to the same method as before: I'd open a screen, screenshot it, send it over, and Claude would read the picture and tell me exactly where to tap next.Claude依然看不到我的Vercel控制台——这一点和上次一样没变。所以我们又用回了同样的办法：我打开一个页面，截图，发过去，Claude看着图片告诉我接下来该点哪里。

The first screenshot — the project's main "Production Deployment" card — looked fine at a glance. But it said the deployment was created 33 minutes earlier. My new article had been pushed only moments before that. Something didn't add up.第一张截图——项目主页的「正式部署」卡片——乍看之下没什么问题。可它显示这个部署是33分钟前创建的。而我的新文章明明是就在那之前不久才推送的。这里对不上。

The second screenshot, the full Deployments list, made it obvious: the newest entries near the top were all marked "Preview," built from feature branches, never promoted. The actual live "Production" tag was sitting on a commit from over 30 minutes earlier. Every push since then — including tonight's T49 article — had gone to GitHub perfectly fine and simply never triggered a new build at all.第二张截图——完整的部署列表——让问题一目了然：列表最上方的最新几条记录全都标着「Preview（预览）」，是从功能分支构建出来的，从未被正式提升上线。真正带着「Production（正式）」标签的，还停留在30多分钟前的那次提交上。从那之后的每一次推送——包括今晚这篇T49文章——都完美无误地送到了GitHub，只是压根没有触发新的构建。

What the screenshots ruled out截图排除掉的可能
  • Not a lost commit — git log confirmed the T49 file was on GitHub's main branch the whole time.不是提交丢失——git log确认T49文件从头到尾都在GitHub的main分支上。
  • Not a typo in the filename or link — the card on tools.html pointed at the exact right address.不是文件名或链接打错字——tools.html上的卡片指向的地址完全正确。
  • Not my code breaking the build — the deployment history showed builds simply weren't being started for new commits, not that they were failing.不是我的代码把构建搞坏了——部署记录显示的是根本没有为新提交启动构建，而不是构建失败。

The Button We Didn't Know Existed一个我们都不知道存在的按钮

Last time (AIT12), the fix was a full migration to a different hosting company — two hours, 191 links to update, backend scripts to rebuild. This time, Claude asked me to look for something smaller first: a "•••" menu on the Deployments screen. Behind it was an option called "Create Deployment" — a way to manually tell Vercel "build this exact commit, right now," bypassing whatever was wrong with the automatic GitHub trigger.上一次（AIT12），修复的办法是彻底搬到另一家托管公司——两个小时，改191个链接，重建后台脚本。这一次，Claude让我先找一个更小的东西：部署页面上的「•••」菜单。点开之后，里面有一个叫「Create Deployment（创建部署）」的选项——可以手动告诉Vercel「现在就构建这个具体的提交」，绕开自动GitHub触发器里出问题的那部分。

I tapped it, tapped the "main" branch chip, and Vercel filled in the very latest commit on its own. One more tap — a big orange button that said "Deploy to Production" — and it started building immediately.我点了它，点了「main」分支的小标签，Vercel自己就把最新的提交填了进去。再点一下——一个写着「Deploy to Production（部署到生产环境）」的大橙色按钮——它立刻就开始构建了。

⚡

The whole fix, start to finish: under 10 minutes of screenshots, one menu we hadn't noticed before, one button tap. No new hosting account, no rebuilt backend scripts, no links to update.整个修复过程，从头到尾：不到10分钟的截图往返，一个我们之前没注意到的菜单，一次按钮点击。不用开新的托管账号，不用重建后台脚本，也不用改任何链接。

The Twist — It Happened Again, Mid-Article意外反转——写这篇文章的当下，同样的事又发生了一次

While I was writing this very story up as a new article, we hit the exact same wall trying to publish it. This time Vercel didn't stay silent — it gave us the real reason in plain text: "Resource is limited — try again in 24 hours (more than 100, code: api-deployments-free-per-day)."就在我把这个故事写成新文章的当下，我们发布这篇文章本身时，又撞上了同一堵墙。这一次Vercel没有沉默——它直接给出了真正的原因：「资源已达上限——请24小时后再试（超过100次，代码：api-deployments-free-per-day）」。

That's not a mystery freeze at all — it's a hard daily cap. Vercel's free plan only allows 100 deployments per day, and between every automated GitHub Action already running on this site (price alerts, daily backups, Tiger refreshes) plus a busy night of Claude sessions pushing fix after fix, we'd simply used them all up. The "Create Deployment" button that fixed T49 wasn't magic — it just happened to land inside that 100-deployment window. The very next attempt, for this article, landed just outside it.这根本不是什么神秘的卡死——而是一个硬性的每日上限。Vercel免费方案每天只允许100次部署，而这个网站本身已经跑着好几个自动化的GitHub Action（价格提醒、每日备份、Tiger数据刷新），再加上今晚Claude连续推送了一整晚的修复，额度很快就被用光了。修好T49的那次「Create Deployment」按钮并不是什么魔法——它只是恰好落在那100次的额度之内。而紧接着为这篇文章再试一次，就恰好落在了额度之外。

🔎

So the honest ending is: T49 (the coin-toss article) went live. This article, describing how we fixed T49, had to wait for the same quota to reset before it could go live itself — a small, fitting irony.所以老实说，结局是这样的：T49（那篇抛硬币文章）成功上线了。而这篇讲述我们如何修好T49的文章，却得等同一个额度重置之后才能自己上线——一个恰如其分的小讽刺。

Why Not Just Switch to Netlify Again?为什么不干脆再换回Netlify？

Because it isn't available right now. My Netlify account's free monthly "operational credits" ran out mid-cycle (see AIT12's update) and won't reset until 16 August 2026 — so it's stuck serving an old, outdated build until then. That ruled out the AIT12 escape route entirely and forced us to find a fix that worked within Vercel itself, which turned out to be the better lesson anyway.因为它现在没法用。我Netlify账号每月的免费「运营额度」提前用完了（见AIT12的更新），要等到2026年8月16日才会重置——所以在那之前它只会停留在一个过时的旧版本上。这就彻底堵死了AIT12里的那条退路，逼着我们必须在Vercel内部找到解决办法——而这反倒是个更好的收获。

What's Different From Last Time和上次相比，哪里不一样

The same bug, a smaller fix.同一个漏洞，更小的修复。

AIT10 said "wait it out." AIT12 said "migrate hosts." This time, Vercel's own dashboard had a manual override built in the whole time — we just hadn't gone looking for it before.AIT10说的是「耐心等」。AIT12说的是「搬家」。这一次，Vercel自己的控制台里其实一直就有一个手动覆盖的选项——只是我们之前没去找过。

Check the deployment list, not just the top card.要看完整的部署列表，而不只是顶部那张卡片。

The single "Production Deployment" summary card looked deceptively fine — "Ready," green dot. Only the full list, showing every recent commit side by side, revealed that new pushes had stopped triggering builds at all.单独那张「正式部署」摘要卡片看起来颇具迷惑性——「Ready」，绿点。只有把完整列表拉出来，把最近每一次提交并排对比，才能发现新的推送根本已经不再触发构建了。

A blocked escape route can force a better answer.一条被堵死的退路，反而能逼出更好的答案。

If Netlify had been available, we might have just switched again instead of digging for the real fix. Being forced to stay and look harder at Vercel is how we found "Create Deployment" in the first place.如果Netlify当时是可用的，我们可能又直接换过去了，而不会去挖真正的解决办法。正因为被逼着留在Vercel里继续深挖，才第一次找到了「Create Deployment」这个选项。

⚠️

Same limit as always: Claude still can't see or click anything on the Vercel dashboard itself — every screenshot and every tap tonight came from my own phone, guided step by step.依然是老规矩里的限制：Claude依然无法自己查看或点击Vercel控制台上的任何东西——今晚每一张截图、每一次点击，都是我自己在手机上一步步照做的。

✅

Now saved as a standing fix: next time a push silently fails to deploy, try Deployments → "•••" → Create Deployment → pick "main" → Deploy to Production first — it's quick and often enough. But if that screen shows a red "Resource is limited... api-deployments-free-per-day" message, stop trying: the daily quota is genuinely used up, and nothing (automatic or manual) will deploy until it resets about 24 hours later. Nothing is broken and nothing is lost — every change is still sitting safely on GitHub, just queued.现在已经记下作为固定解法：下次再遇到推送悄悄没能部署上线，先试试 Deployments → 「•••」→ Create Deployment → 选「main」→ Deploy to Production——这一步又快又常常管用。但如果那个画面出现红色的「Resource is limited... api-deployments-free-per-day」提示，就别再试了：这是每日额度真的用完了，在大约24小时后重置之前，不管是自动还是手动都无法再部署。什么都没坏、什么都没丢——所有改动依然安全地留在GitHub上，只是在排队等待而已。
