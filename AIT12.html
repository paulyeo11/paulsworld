AIT12: The Night We Finally Left Vercel —
A Live Migration From My Phone AIT12：我们终于离开Vercel的那一晚——
一次在手机上完成的现场迁移

📅 July 20262026年7月 ⏱️ 8 min read8分钟阅读

Two hours. One SPY chart request that turned into a full website migration, all steered from my phone in bed. This is the story of the night the old bug came back — and this time, instead of waiting for it to fix itself, we changed hosts entirely. 两个小时。一个SPY图表的小请求，最后变成了一次完整的网站迁移，而且全程都是我在床上用手机操作的。这是那个老毛病再次发作的一晚——这一次，我们没有等它自己恢复，而是彻底换了服务商。

It Started With a Chart一切从一张图表开始

I sent Claude a screenshot of my NVDA chart on the T29 page and asked for the same thing for SPY. Simple enough. Then I saw a screenshot from Bang, my options trading mentor — a proper price chart with a shaded "Profit Zone" band — and asked for that style too. Claude built both: a strike "rail" showing where price sits between my put and call strikes, and underneath it, a real 6-month price line with the profit zone highlighted in green, matching Bang's format almost exactly.我给Claude发了T29页面上NVDA图表的截图，让它为SPY也做一个。挺简单的。后来我看到我的期权交易导师Bang发的一张截图——一张带有阴影「盈利区」色带的正规股价图，于是也让它照做。Claude把两样都做出来了：一条显示股价落在我认沽与认购行使价之间位置的「导轨」，下面还有一条真实的6个月股价线，盈利区用绿色高亮显示，几乎完全照搬了Bang的格式。

Except when I opened it on my phone, the chart was empty. "Live data unavailable — check IBKR directly."可是当我在手机上打开它时，图表却是空的。「实时数据不可用——请直接查看IBKR」。

The Same Ghost, Back Again同一个幽灵，又回来了

Long-time readers of this series will recognise this one. Back in AIT10, my site's connection to Vercel — the company that actually publishes my pages live — froze for about an hour and then quietly fixed itself. That article's advice was: don't panic, wait 30–60 minutes, it'll probably sort itself out.看过这个系列的读者可能会觉得眼熟。早在AIT10那篇文章里，我网站与Vercel（真正负责把我的网页发布上线的公司）之间的连接曾卡死了大约一个小时，然后自己悄悄恢复了正常。那篇文章给的建议是：别慌，先等30到60分钟，它多半会自己恢复。

This time, it didn't. Claude fixed the chart's actual data source — turned out it was calling Yahoo Finance directly from my browser instead of through the site's own working proxy, which browsers block. That got fixed, pushed, confirmed on GitHub. Still blank on my phone. Then a caching bug in my site's own offline-support code, which was quietly serving back an old failed answer instead of trying again. Fixed that too, pushed again. Still blank.这一次没有。Claude修好了图表真正的数据来源——原来它是直接从我的浏览器调用雅虎财经，而不是通过网站自己那条本来就能用的代理线路，浏览器会拦截这种直接调用。这个问题修好了，推送了，也在GitHub上核实过。可我手机上看到的还是空白。接着又发现是网站自己的离线支持代码里有个缓存漏洞，一直悄悄把之前失败的旧结果拿出来用，而不是重新尝试。这个也修好了，又推送了一次。结果还是空白。

Every fix was real. Every commit landed safely on GitHub. And every single time, my phone showed me the exact same broken page — as if none of it had ever happened.每一次修复都是真实有效的。每一次提交都安全地落在了GitHub上。可每一次，我手机上显示的都是同一个坏掉的页面——好像什么都没发生过一样。

Chasing It Through My Phone用手机一路追查

This is where it got interesting. Claude can't see the Vercel dashboard — only I can, on my screen. So we did this the hard way: I'd tap something, screenshot it, send it over; Claude would read the screenshot and tell me exactly what to tap next. Deployments list. Status filters. A "Canceled" checkbox hiding behind a date picker I opened by accident. Round after round.这就是有意思的地方了。Claude看不到Vercel的控制台——只有我自己的屏幕能看到。所以我们只能用最笨的办法：我点一下，截图，发过去；Claude看着截图告诉我接下来该点哪里。部署列表、状态筛选器、藏在一个我不小心打开的日期选择器后面的「已取消」勾选框。就这样一轮又一轮。

Eventually the picture became clear: the live production deployment was still sitting on a commit from over an hour earlier. Every fix after that point had been pushed cleanly to GitHub — and simply never showed up in Vercel's own deployment list at all. Not failed. Not queued. Just... never arrived.最终真相渐渐清晰：正式上线的部署仍然停留在一个多小时前的提交上。那之后的每一次修复都干净利落地推送到了GitHub——但在Vercel自己的部署列表里，却压根没有出现过。不是失败，也不是排队等待，就是……根本没有到达。

What we ruled out, again我们再次排除的可能
  • GitHub had every commit — verified directly through GitHub's own API each time.GitHub上确实有每一次提交——每次都直接通过GitHub自己的API核实过。
  • My code wasn't broken — the same fix, deployed fresh to a brand-new host later that night, worked immediately.我的代码没有问题——同样的修复，当晚稍后部署到一个全新的服务商上，立刻就生效了。
  • It wasn't a browser caching illusion — we tested with a URL that had never been requested before, guaranteed impossible to cache. Same stale result.也不是浏览器缓存造成的错觉——我们用了一个此前从未请求过、绝不可能被缓存的网址来测试。结果还是一样的陈旧数据。

"Every Day I Encounter This"「每天我都会遇到这个问题」

That's what I told Claude, and it changed everything. AIT10 was written as if this was a one-off storm to wait out. It isn't. It's a pattern. And once something is a pattern instead of an accident, "wait for it to pass" stops being good advice.这是我对Claude说的话，而这句话改变了一切。AIT10那篇文章写的时候，还以为这只是一次偶发的风暴，等过去就好。但事实并非如此，这是一种反复出现的模式。而一旦某件事从「意外」变成了「模式」，「等它过去」就不再是好建议了。

So I said it plainly: don't use Vercel. Let's try somewhere else.于是我直接说了：别用Vercel了，我们换个地方试试。

Building the Escape Route搭建退路

We picked Netlify — similar free tier, similar auto-deploy-from-GitHub setup, and it turned out I already had an account from an old side project. Claude walked me through connecting my site's GitHub repository step by step: sign in, authorize, pick the repository (I grabbed the wrong one at first — easy mistake, one tap to fix), configure, deploy.我们选择了Netlify——免费额度差不多，从GitHub自动部署的方式也差不多，而且发现我早年做一个副业项目时就已经注册过账号。Claude一步步带着我把网站的GitHub仓库连接了起来：登录、授权、选择仓库（我一开始选错了一个——很容易犯的小错，点一下就改过来了）、配置、部署。

⚡

The moment that told us everything: the very first Netlify deploy finished in under two minutes. No stuck queue, no missing commits, no mystery. It just worked — the first time.那一刻说明了一切：Netlify的第一次部署，两分钟不到就完成了。没有卡住的队列，没有失踪的提交，没有任何谜团。就是好用——第一次就好用。

From there it snowballed into a proper migration: every hardcoded link on my site — 191 pages' worth — pointing at the old address got switched to the new one. The three small backend scripts my site relies on (the live price feed, and two AI helper functions) got rebuilt for Netlify's slightly different format. My own working notes for Claude got updated so this becomes the new default, not just tonight's workaround.从那以后，事情自然而然发展成了一次正式的迁移：网站上所有指向旧地址的硬编码链接——总共191个页面——全部改成了新地址。网站依赖的三个小型后台脚本（实时价格数据，以及两个AI辅助功能）也按Netlify略有不同的格式重新搭建了一遍。我给Claude留的工作笔记也更新了，这样从今晚起这就是新的默认做法，而不只是权宜之计。

One Migration Led to Another一次迁移，牵出另一次

Feeling good about it, I asked Claude to do the same for my separate investment tracking site. That's when we found the migration's first real casualty: a card on that site's homepage linked out to a page that no longer existed — a leftover from months ago, nothing to do with tonight, just never caught until we went looking. Then a second one, a broker link pointing at a filename that had been renamed ages back.迁移顺利，我心情不错，就让Claude把我另一个投资追踪网站也照样处理一下。就是在这时，我们发现了这次迁移中真正的「战损」：那个网站首页上的一张卡片，链接指向了一个早已不存在的页面——是几个月前留下的旧账，跟今晚的事毫无关系，只是一直没被发现，直到我们主动去查。接着又发现第二个：一个经纪商链接指向的文件名，早就已经被改过了。

We fixed both, right there on my phone, editing the file directly on GitHub's website since Claude couldn't reach that particular repository itself. Double-tap a word, type the replacement, screenshot to confirm, commit. Slow, but it worked — and it caught bugs that had been quietly broken for months.我们当场就在手机上把两个都修好了——直接在GitHub网站上编辑文件，因为Claude自己没办法访问那个仓库。双击选中一个词，输入替换的内容，截图确认，提交。虽然慢，但确实管用——而且还顺带揪出了几个已经悄悄坏掉好几个月的漏洞。

🔧

By the end of the night, both sites were live on the new host, both broken links were fixed, and a third batch of new cards was ready and waiting for me to upload the next time I'm at the keyboard, not half-asleep on my phone.到那晚结束时，两个网站都已经在新的服务商上线，两个失效链接都修好了，还有第三批新卡片已经准备就绪，等我下次坐在电脑前而不是半睡半醒地拿着手机时再上传。

Full Circle — The Best of Both Worlds兜了一圈——鱼与熊掌兼得

A few hours after the Netlify migration went live, Vercel quietly started working again — deployments picked up normally, no freeze, no missing commits. So that same night, I asked Claude to move the live site back to Vercel.Netlify迁移上线几个小时后，Vercel竟悄悄恢复正常了——部署恢复如常，没有卡住，也没有丢失的提交。于是同一晚，我让Claude把正式网站搬回了Vercel。

That switch back took only a few minutes — because nothing about the Netlify side had to be undone. The same GitHub repository was already connected to both hosts; flipping which one served the live site was just a matter of pointing the domain back. Netlify's `netlify.toml` and its backend functions were left in place exactly as built, wired and ready, instead of being torn out.这次切回只花了几分钟——因为Netlify那边什么都不用拆。同一个GitHub仓库本来就同时连接着两个平台，切换哪一个来提供正式网站，只是把域名指回去而已。Netlify的`netlify.toml`配置和后台函数都原封不动地留着，接好线，随时待命，而不是被拆掉。

🌉

This is the "best of both worlds" part: Vercel stays the main, familiar home for the site. But Netlify is no longer a one-off rescue built from scratch at 9pm — it's a live, tested, already-working twin sitting quietly in the background. If Vercel ever freezes again, the fix isn't a two-hour phone migration anymore. It's flipping a switch that's already been tested to work.这就是「鱼与熊掌兼得」的地方：Vercel依然是网站主要、熟悉的家。但Netlify不再是晚上9点临时从零搭建的救急方案——它是一个已经上线、经过测试、随时待命的「孪生兄弟」，安安静静地待在背后。如果Vercel哪天再次卡住，需要做的不再是耗时两小时的手机大迁移，而只是切换到一个早已验证可用的备用方案。

What I'd Tell Myself Next Time如果重来一次，我会这样提醒自己

A bug that happens once is bad luck. The same bug happening again is information.一次性的漏洞是运气不好。同一个漏洞再次发生，那就是一种信号。

AIT10's "just wait it out" was the right call the first time. The second time the same thing happened, that advice needed to be retired, not repeated.AIT10那篇「耐心等就好」的建议，在第一次发生时是对的。但第二次同样的事情再次发生时，这条建议就该被放弃，而不是重复使用。

Debugging entirely by phone screenshot is slow but genuinely works.完全靠手机截图来排查问题，虽然慢，但确实行得通。

Every dead-end and mistap along the way still added up to the right answer. I didn't need my laptop for any of this.过程中每一次走错路、点错按钮，最终都汇聚成了正确的答案。这整件事我完全没有用到笔记本电脑。

A migration is a good excuse to check everything nearby.一次迁移，正好是检查周边一切的好机会。

Neither of the two broken links we fixed tonight were caused by the move to Netlify — they'd been quietly broken for a while. We only found them because we went looking while something else was already open.今晚修好的那两个失效链接，其实都不是因为搬到Netlify才出问题的——它们早就悄悄坏掉一段时间了。只是因为我们当时正好在检查别的东西，才顺带把它们揪了出来。

Some things genuinely need to wait until I'm at my desk.有些事真的得等我坐在书桌前才能做。

Claude built the next round of changes and even the finished file, ready to go — but couldn't reach that second repository to upload it itself. Knowing when to stop for the night, mid-task, was the right call at 10:40pm.Claude已经把下一批更改，甚至连做好的文件都准备好了——但没办法自己访问那第二个仓库把文件传上去。晚上10点40分，在任务做到一半时知道该收工休息了，这个判断是对的。

⚠️

One important limit, same as last time: Claude still cannot see a dashboard it isn't given direct access to, or click buttons on my screen for me. Every Vercel screenshot and every GitHub edit tonight still had to come from my own hands.一个重要的限制，和上次一样：Claude依然无法查看没有被授予直接访问权限的控制台，也无法替我点击屏幕上的按钮。今晚每一张Vercel的截图、每一次GitHub编辑，最终都得靠我自己动手完成。

✅

The reassuring part, also same as last time: nothing was ever lost. Every fix was safely saved on GitHub the entire time, on both sites, through every dead end. The only thing that took two hours was finding the door — not rebuilding the house.让人安心的一点，同样和上次一样：没有任何东西丢失。每一次修复，在两个网站上，在每一次走进死胡同的过程中，始终都被安全保存在GitHub上。花了两个小时的，只是找到那扇门——而不是重建整栋房子。
